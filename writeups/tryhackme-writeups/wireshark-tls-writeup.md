# Wireshark TLS Decryption — Login Credentials Challenge
### TryHackMe Guided Challenge Writeup

> This is a guided challenge writeup from TryHackMe completed as part of my learning journey.
> All activities were performed in a controlled lab environment.

---

## Objective
Decrypt encrypted TLS network traffic using Wireshark and locate hidden login credentials inside a captured packet.

---

## Background

Normally, TLS encrypts all traffic between a client and server making it unreadable to anyone capturing packets. However, if the TLS session keys are exposed, all encrypted traffic can be decrypted and read in plain text.

In this challenge, Chromium was configured to log its TLS session keys to a file. This simulates a scenario where an attacker has gained access to a system's TLS keys and can now decrypt all captured traffic.

---

## Tools Used
- **Wireshark** — packet capture and analysis tool
- **Chromium** — browser configured to dump TLS keys
- **randy-chromium.pcapng** — packet capture file containing the encrypted traffic
- **ssl-key.log** — TLS session keys dumped by Chromium

---

## Step 1 — Capture Traffic with TLS Key Logging

Chromium was launched with a special flag to dump all TLS session keys:

```bash
chromium --ssl-key-log-file=~/ssl-key.log
```

This logs every TLS session key to `ssl-key.log` in the home directory. These keys can later be used by Wireshark to decrypt the captured traffic.

---

## Step 2 — Open the Packet Capture in Wireshark

- Navigated to the `Documents` folder
- Opened `randy-chromium.pcapng` in Wireshark
- At this point all TLS traffic appeared as encrypted gibberish

---

## Step 3 — Load the TLS Key Log into Wireshark

To decrypt the TLS traffic:

1. Right clicked any **TLS packet** in Wireshark
2. Selected **Protocol Preferences**
3. Selected **Transport Layer Security**
4. Clicked **Open Transport Layer Security preferences**
5. Clicked **Browse** and located `ssl-key.log` in the Documents directory
6. Clicked **OK**

✅ All TLS traffic was now fully decrypted and readable in plain text.

---

## Step 4 — Locate the Login Credentials

To find the login credentials, used Wireshark's built-in search function:

1. Pressed **`Ctrl + F`** to open the search bar
2. Changed the first dropdown to **Packet details**
3. Changed the second dropdown to **String**
4. Typed **`pass`** and pressed **Enter**

Wireshark jumped directly to the packet containing:
```
HTML Form URL Encoded: application/x-www-form-urlencoded
```
This indicated the packet contained data submitted from a login form.

---

## Step 5 — Extract the Credentials

Expanded the `HTML Form URL Encoded` section of the packet by clicking the 
arrow next to it in Wireshark, which revealed the following form data in plain text:

| Field | Value |
|---|---|
| **email** | `strategos@networking.thm` |
| **pass** | `THM{B8WM6P}` |
| login_source | comet_headerless_login |
| login | 1 |

**Flag: `THM{B8WM6P}`** 🚩

---

## Breaking Down the Packet

```
HyperText Transfer Protocol 2
  └── HTML Form URL Encoded: application/x-www-form-urlencoded
        ├── jazoest = "2877"          → hidden security token (CSRF protection)
        ├── lsd = "AVpeRE3H6tE"       → hidden security token auto-generated by site
        ├── email = "strategos@networking.thm"   → username in plain text ⚠️
        ├── pass = "THM{B8WM6P}"      → password in plain text ⚠️
        ├── login_source = "comet_headerless_login"  → internal login identifier
        ├── next = ""                 → redirect destination after login
        └── login = "1"              → confirms this is a login request
```

**`application/x-www-form-urlencoded`** — means this packet contains data submitted from an HTML login form. When a user clicks "Login" on a website, this is what gets sent to the server.

**`jazoest` and `lsd`** — hidden security tokens automatically generated by the website to prevent fake or forged form submissions (CSRF tokens). Not relevant to our investigation.

**`email`** — the username/email entered in the login form, fully visible in plain text once TLS was decrypted.

**`pass`** — the actual password entered, also completely visible in plain text once TLS was decrypted.

**`login_source`** — tells the server internally where the login request originated from.

## What I Learned

- TLS encrypts traffic in transit but if session keys are **leaked or exposed**, all traffic can be fully decrypted
- Login credentials inside HTTP POST requests are completely readable once TLS is decrypted
- This is why **TLS key log files should never be exposed** on a real system
- HTTPS protects data **in transit** but does not protect against key exposure
- Tools like Wireshark are powerful for both **network troubleshooting** and **security analysis**
- Unencrypted or decrypted credentials in network traffic is a critical security risk in real environments

---

## Mitigation

To protect against this type of attack in a real environment:
- Never expose TLS key log files on production systems
- Use **certificate pinning** to prevent key extraction
- Monitor for unusual SSL/TLS key logging activity on endpoints
- Ensure **endpoint security** to prevent attackers from accessing key files
